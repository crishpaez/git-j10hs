Working Directory

El area de trabajo corresponde a la carpeta donde inicializamos al repositorio de git, rara vez vamos a estar utilizando la linea de comandos y es basicamente donde vamos a crear, editar y eliminar los diferentes archivos de nuestro proyecto.

  • dir || ls :Muestra el contenido de la carpeta actual
  • mkdir <folder name> :Crea una carpeta con el nombre especificado.
  • cd <path> :Cambia la carpeta actual por la seleccionada.
  • git init :Inicializa un repositorio de git en la carpeta actual.
  • rm -rf .git :Elimina el repositorio de manera forzada

Staging Area (INDEX)

El area de control de cambios es aquella donde agregamos los archivos que vamos a estar haciendoles seguimiento, cada vez que agregamos un elemento al INDEX, se realiza una captura de pantalla del codigo (SNAPSHOT) que luego debera ser confirmada para determinar su permanencia en el historial de cambios.

  • git status muestra el estado de los archivos del Working Directory con respecto al INDEX, dentro de los estados mas comunes estan:
    • untracked: no se esta realizando el seguimiento del archivo, irrecuperable en caso de perdida
    • staged: se realiz√≥ la captura del codigo, pero todavia no se confirmaron los cambios.
    • commited: los cambios realizados en el archivo se encuentran almacenados en el historial
    • modified: el archivo difiere con respecto a lo registrado en el INDEX o la ultima confirmacion
  • git add <file> agrega al INDEX el conjunto de archivos seleccionados
  • git diff compara los archivos del Working Directory con lo que se registra en el INDEX.
  • git restore --staged <file> elimina la ultima captura del INDEX, no modifica el archivo.
  • git <file> elimina el archivo del INDEX, no se realiza mas seguimientos.

Repository (LOCAL/REMOTO)

Es el area donde se almacenan todas las confirmaciones de cambios en archivos de tipo BLOB(Binary Large Object File) distribuidos en una estructura de carpetas identificadas de manera hexadecimal que van desde el 00 al FF. Esto nos permite crear puntos de control a los que podemos volver en cualquier momento.

  • git commit abre el editor predeterminado para realizar la confirmacion de cambios.
  • git log muestra el historial de cambios ordenados de manera descendente
  • git blame <file> muestra los autores y confirmaciones de cada linea de un archivo.
  • git push <remote> <branch> agrega los cambios de la rama al repositorio remoto.
  • git fetch <remote> captura el historial de cambios del repositorio remoto.

Apuntadores de GIT

HEAD

Apuntador movil que indica al usuario en que punto del historial se encuentra, normalmente suele apuntar a la ultima confirmacion, pero podemos utilizarlo para desplazarnos dentro del historial de cambios. Tambien se utiliza como referente para otros comandos como checkout, diff, etc... Si escribimos HEAD~2 o HEAD^^ estamos referenciando el commit que se encuentra 2 confirmaciones atras.

BRANCH

Apuntador dinamico que siempre se corresponde con el ultimo commit de la historia, se utiliza para facilitar la navegacion dentro del historial de confirmaciones y para probar caracteristicas, corregir errores o realizar revisiones sin necesidad de afectar la historia principal.
